# PS Transfer Service

Отдельный микросервис переводов.

## Ответственность

- хранение карточных переводов и переводов между своими счетами;
- публичный API предварительного расчета и создания переводов;
- история и статусы переводов;
- transactional outbox для команд исполнения;
- получение результата исполнения через Kafka.

Балансами, картами, счетами и валютными курсами продолжает владеть основной
backend. Поэтому авторитетная проверка доступности средств, карт и счетов,
получение актуальных курсов и изменение балансов выполняются через core.
Расчет конвертации, комиссии и суммы списания выполняет `ps-transfer`, который
также владеет жизненным циклом перевода, историей, статусом и доставкой команды
исполнения.

Карточный перевод исполняется асинхронно:

1. `ps-transfer` сохраняет перевод со статусом `IN_PROGRESS` и outbox-событие.
2. Команда публикуется в `ps-transfer-execute`.
3. Основной backend атомарно блокирует и изменяет оба счета.
4. Результат публикуется в `ps-transfer-result`.
5. `ps-transfer` меняет статус на `SUCCESS` или `FAILED`.

## Локальный запуск

```powershell
docker compose up -d
mvn spring-boot:run
```

Порты:

- API: `8082`;
- PostgreSQL: `5434`;
- основной backend: `8080`;
- BFF: `9090`;
- Kafka: `9092`;
- Keycloak: `8081`.

Перед запуском микросервиса должны быть доступны Kafka, Keycloak и основной
backend. После изменения маршрутов BFF его также необходимо перезапустить.

В production задайте одинаковое секретное значение:

```text
CORE_INTERNAL_TOKEN=<secret>       # ps-transfer
TRANSFER_INTERNAL_TOKEN=<secret>   # основной backend
```

Оно защищает технический `/api/v1/banking-operation-support/**` от прямого
вызова через BFF.

Liquibase создает актуальную схему и загружает начальную историю из
`db/changelog/insert-data.sql`. Отдельный backfill не требуется.
